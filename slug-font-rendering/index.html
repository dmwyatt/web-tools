How Slug Renders Fonts on the GPU

An interactive explainer of Eric Lengyel's Slug algorithm

1. The Bitmap Problem

Why games and GPU applications need a special font rendering algorithm.

Bitmap (how your browser does it)
Vector curves (what Slug uses)
1x

Your browser renders text by converting mathematical font outlines into a grid of colored pixels. At the intended size, this looks great. But zoom in and you see the pixel grid.

In a 3D game, a street sign might be 5 pixels tall in the distance and 500 pixels tall up close. Pre-rendering bitmaps at every possible size wastes enormous amounts of memory. Rendering them on the CPU each frame is too slow.

The Slug algorithm takes a different approach: store the font's mathematical curves and evaluate them directly on the GPU, per pixel, in real time. The result is always crisp, at any size.

2. What's Inside a Font?

Fonts are not pictures of letters. They are mathematical recipes for drawing curves.

A font file stores each letter as a series of quadratic Bezier curves. Each curve has three control points: a start (filled dot), an end (filled dot), and a "pull" point (hollow dot) that bends the curve toward it.

Try dragging the control points above to reshape the letter. This is literally what font design software does.

TrueType fonts (the most common format) use exactly these quadratic curves. The Slug algorithm sends these control points to the GPU and evaluates the curves mathematically for every pixel on screen.

3. Inside or Outside?

How the GPU decides whether each pixel is part of the letter.

3a. Ray Casting

The GPU needs to answer one question for every pixel: "Am I inside or outside the letter?" It answers this by casting a ray and counting how many times it crosses the outline.

Click anywhere to test a point
Winding number: 0 outside

A horizontal ray is cast from the test point to the right. Each time it crosses a curve going upward, add 1. Going downward, subtract 1. If the total (the "winding number") is not zero, the pixel is inside the letter.

This is the nonzero winding rule, and it is the default fill rule Slug uses.

3b. Anti-aliasing with Coverage

Sharp inside/outside decisions produce jagged edges. Slug uses two rays (horizontal and vertical) and measures how far each crossing is from the pixel center to compute partial coverage.

No anti-aliasing
With coverage-based AA

Pixels near the edge get a fractional coverage value (e.g., 0.3 means 30% covered). This produces smooth, anti-aliased edges without any blur. The two rays are combined using distance-based weights.

3c. Band Optimization

Testing every curve for every pixel would be wasteful. Slug divides the glyph into horizontal and vertical bands.

Each pixel only tests the curves in its band, not every curve in the glyph. For the letter "R" shown above, each band contains only 3 to 8 curves instead of all 23. This is what makes Slug fast enough for real-time rendering.
